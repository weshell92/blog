## 从一个真实场景说起

最近在写博客系统的时候，我发现好几个函数都需要做「执行时间统计」，最笨的办法是每个函数里都加上：

```python
import time
start = time.time()
# ... 函数逻辑
print(f"耗时：{time.time() - start:.3f}s")
```

这样代码重复，而且把「计时」这个关注点混入了业务逻辑。于是我想到了装饰器。

## 装饰器的本质

装饰器本质上是**高阶函数**——接收一个函数作为参数，返回一个新函数。

```python
import time
from functools import wraps

def timer(func):
    @wraps(func)  # 保留原函数的元信息
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        elapsed = time.time() - start
        print(f"[{func.__name__}] 耗时：{elapsed:.3f}s")
        return result
    return wrapper

@timer
def load_all_posts():
    # 加载所有文章...
    pass
```

`@timer` 只是语法糖，等价于 `load_all_posts = timer(load_all_posts)`。

## 带参数的装饰器

更进一步，如果想控制「只有耗时超过阈值才打印」：

```python
def timer(threshold=0.1):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            start = time.time()
            result = func(*args, **kwargs)
            elapsed = time.time() - start
            if elapsed > threshold:
                print(f"[{func.__name__}] 慢查询：{elapsed:.3f}s")
            return result
        return wrapper
    return decorator

@timer(threshold=0.5)
def slow_function():
    pass
```

三层嵌套，但逻辑很清晰。

## 我的感悟

装饰器体现了**关注点分离**的设计思想：业务逻辑归业务逻辑，横切关注点（日志、计时、缓存、权限）用装饰器处理。

Streamlit 的 `@st.cache_data` 本身也是装饰器，这让我更直观地理解了它为什么能「神奇地」缓存函数结果。

> 好的抽象不是消除复杂度，而是把复杂度放到它该在的地方。
